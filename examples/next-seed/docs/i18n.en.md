# Internationalization (i18n) in `next-seed`

**中文:** [i18n.md](./i18n.md)

This document describes how i18n is layered and wired in `next-seed`, and recommended usage under the **App Router** and **Pages Router**. The stack builds on [next-intl](https://next-intl-docs.vercel.app/) plus project conventions: **identifier layer → mapping layer → application layer**.

---

## 1. Core concepts

| Concept                                       | Role                                                                                                                                                                                                                                  |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Messages (dictionary)**                     | Key–value map for `next-intl`. Keys are namespace-prefixed strings (e.g. `common:save`, `api:user__not_found`).                                                                                                                       |
| **Default namespaces**                        | `defaultNamespaces: ['common', 'api']` in `shared/config/i18n.ts`. The root layout passes **only these namespaces** (by default) into the client `NextIntlClientProvider`—additional page namespaces can be merged in nested layouts. |
| **`PageI18nInterface`**                       | Shape for page-level copy / SEO in `shared/config/i18n-mapping/PageI18nInterface.ts`. Concrete mappings (e.g. `homeI18n`) store **message keys** as field values; after translation they become display strings.                      |
| **`PageI18nProvider` + `usePageI18nMapping`** | Pushes the **fully resolved** mapping object for the current page into React context so children read plain strings without calling `t()`.                                                                                            |

---

## 2. End-to-end examples (three scenarios)

One linear story: **identifier constants (keys) → mapping object (values = keys) → page or component consumes**. Sample code below—replace keys and paths with your real page.

### 2.1 App Router page

**Flow:** `generateLocales` writes `public/locales` → on the server, `getI18nInterface(locale, mapping, namespace)` produces translated object `tt` → `<PageI18nProvider value={tt}>` → client children use `usePageI18nMapping()` and read `tt.title`, etc.

```tsx
// --- shared/config/i18n-identifier/pages/page.example.ts
// Export message keys (after running tools/generateLocales they appear in en.json / zh.json)

// --- shared/config/i18n-mapping/ExamplePageI18n.ts
import {
  EXAMPLE_SAVE_KEY,
  EXAMPLE_TITLE_KEY
} from '../i18n-identifier/pages/page.example';

export const examplePageI18nNamespace = 'page_example';

export const examplePageI18n = Object.freeze({
  title: EXAMPLE_TITLE_KEY,
  saveButton: EXAMPLE_SAVE_KEY
});

export type ExamplePageI18nType = typeof examplePageI18n;

// --- src/app/[locale]/example/page.tsx (Server Component)
import { PageI18nProvider } from '@/uikit/context/PageI18nContext';
import {
  getI18nInterface,
  getLocale,
  type PageParamsType
} from '@server/render/pageRouteParams';
import {
  examplePageI18n,
  examplePageI18nNamespace
} from '@config/i18n-mapping/ExamplePageI18n';

export default async function ExamplePage({
  params
}: {
  params: Promise<PageParamsType>;
}) {
  const { locale: localeParam } = await params;
  const locale = getLocale({ locale: localeParam });
  const tt = await getI18nInterface(
    locale,
    examplePageI18n,
    examplePageI18nNamespace
  );

  return (
    <PageI18nProvider value={tt}>
      <ExamplePanel />
    </PageI18nProvider>
  );
}

// --- ExamplePanel.tsx (Client Component)
('use client');

import { usePageI18nMapping } from '@/uikit/context/PageI18nContext';
import type { ExamplePageI18nType } from '@config/i18n-mapping/ExamplePageI18n';

export function ExamplePanel() {
  const tt = usePageI18nMapping<ExamplePageI18nType>();
  return (
    <>
      <h1>{tt.title}</h1>
      <button type="button">{tt.saveButton}</button>
    </>
  );
}
```

**Takeaway:** The root layout only supplies `common` + `api` to the client by default. Page-specific keys are resolved on the server into `tt`; children do not rely on `page_*` being present in `NextIntlClientProvider`.

### 2.2 Pages Router page

**Flow:** In `getStaticProps` / `getServerSideProps`, **`new PagesRouteParams(params).getI18nMessages('page_example')`** → yields `common` + `api` + page namespace → returned as **`props.messages`** → `_app` passes them to `NextIntlClientProvider` → the page (client) uses **`useI18nMapping(examplePageI18n)`** for `tt`, or optionally **`getI18nInterface` in props** to fix copy entirely at build/request time.

```tsx
// --- getStaticProps (excerpt)
import { PagesRouteParams } from '@server/render/PagesRouteParams';
import type { PagesRouteParamsType } from '@server/render/PagesRouteParams';
import type { GetStaticPropsContext } from 'next';

const namespace = 'page_example';

export async function getStaticProps({
  params
}: GetStaticPropsContext<PagesRouteParamsType>) {
  const pageParams = new PagesRouteParams(params);
  const messages = await pageParams.getI18nMessages(namespace);

  return {
    props: { messages }
  };
}

// --- src/pages/[locale]/example.tsx (Client page)
('use client');

import { useI18nMapping } from '@/uikit/hook/useI18nMapping';
import { examplePageI18n } from '@config/i18n-mapping/ExamplePageI18n';

export default function ExamplePage() {
  const tt = useI18nMapping(examplePageI18n);
  return <h1>{tt.title}</h1>;
}
```

**Optional (more static):** In `getStaticProps`, `await pageParams.getI18nInterface(examplePageI18n, namespace)`, put `tt` in `props`, and render without `useI18nMapping`.

### 2.3 Ordinary components (both routers)

| Scenario                                              | Approach                                                                                                                                      |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Copy lives under **`common` / `api`**                 | In a client component: **`const t = useWarnTranslations(); return t(SOME_KEY);`** (matches the default messages injected at the root layout). |
| Copy is **page mapping already resolved by the page** | Component must sit under **`PageI18nProvider`** and use **`usePageI18nMapping<T>()`**.                                                        |

```tsx
'use client';

import { useWarnTranslations } from '@/uikit/hook/useWarnTranslations';
// Key from identifier, under common / api namespace
import { SOME_COMMON_KEY } from '@config/i18n-identifier/common/common';

export function SaveChip() {
  const t = useWarnTranslations();
  return <span>{t(SOME_COMMON_KEY)}</span>;
}
```

(`SOME_COMMON_KEY` is a placeholder—replace with a real export from `i18n-identifier` under `common`.)

**Note:** `useWarnTranslations` / `useI18nMapping` run only on the **client** and require **`NextIntlClientProvider`** to expose the right `messages`. On the **server**, use **`getI18nInterface`** and pass strings via props or `PageI18nProvider`.

## 3. Three-layer architecture

### 3.1 Layer 1: Identifiers (`shared/config/i18n-identifier`)

- Holds **stable message keys**, usually split across `.ts` modules.
- Generated JSON entries follow **`namespace:key`** (per `ts2locales` rules).
- **Do not hand-edit** `public/locales/*.json`; regenerate from identifiers.

### 3.2 Layer 2: Mappings (`shared/config/i18n-mapping`)

- **Human-friendly objects** such as `homeI18n`, `adminRequestLogs18n`.
- **Field values** reference identifier exports (i.e. **next-intl message keys**).
- Types and comments (e.g. `HomeI18nInterface`) document fields for that page/module.

### 3.3 Layer 3: Application

Pick one or combine:

1. **`useWarnTranslations()`** — Enhanced `t` for key-by-key translation; wraps `next-intl`’s `useTranslations` with missing-key warning / fallback behavior.
2. **`useI18nMapping(mappingObject)`** — On the **client**, turns a mapping whose values are keys into a fully translated object; good for Pages Router or when you skip server-side resolution.
3. **`getI18nInterface(locale, mapping, namespace?)` (server)** — Resolves the mapping on the **server** for `generateMetadata`, Server Components, or passing into `PageI18nProvider`.
4. **`usePageI18nMapping<T>()`** — Reads the resolved object from a parent **`PageI18nProvider`** (`T` should align with `PageI18nInterface` conventions).

## 4. Generating and loading messages

### 4.1 Generation: `tools/generateLocales.ts`

- Default scan: all `.ts` under `shared/config/i18n-identifier`.
- Output: `public/locales/{{lng}}.json` (e.g. `en.json`, `zh.json`); languages from `i18nConfig.supportedLngs`.
- Run from build/local scripts; whether JSON is committed is a team policy choice.

### 4.2 Loading: `src/i18n/loadMessages.ts`

- **By default** reads bundled `public/locales/{locale}.json` when `useApiLocales === false`.
- If **`useApiLocales`** is `true` in `shared/config/common.ts`, tries `SITE_URL/api/locales/json?locale=...`, then falls back to local JSON.

### 4.3 App Router server config: `src/i18n/request.ts`

- `getRequestConfig` calls **`loadMessages(locale)` for the full bundle** (no namespace filter per call).
- Server helpers such as `getTranslations` can therefore resolve **any key present in JSON** for that request.

### 4.4 Client-visible messages: root `layout` filtering

- `src/app/[locale]/layout.tsx` uses **`getI18nMessages(locale)`** (see `server/render/pageRouteParams.ts`): takes `getMessages`, then filters to **`i18nConfig.defaultNamespaces` (`common` + `api`)** before `NextIntlClientProvider`.
- **Implication:** Client `useWarnTranslations()` / `useI18nMapping` for **page-only namespaces** (e.g. `page_home`) need those namespaces in the provider—**merge in a layout** or use **`PageI18nProvider`** with server-resolved copy.

## 5. App Router recommendations

### 5.1 Shared dictionary (root layout)

- `NextIntlClientProvider` receives **default `common` + `api`** for global components and `I18nService.t`.

### 5.2 Page-level: resolve on the server, pass down (home pattern)

Typical steps (see `src/app/[locale]/page.tsx`):

1. `getLocale(resolvedParams)` for the active locale.
2. **`const tt = await getI18nInterface(locale, pageMapping, pageNamespace)`** for the full translated object.
3. In **`generateMetadata`**, use **`await getI18nInterface(locale, pageMapping)`** for SEO (pass the third argument if metadata needs a specific namespace).
4. Wrap the page’s client subtree with **`<PageI18nProvider value={tt}>`**; deep client components use **`usePageI18nMapping<YourPageI18nType>()`**.

**Benefit:** Page copy resolves once on the server; the client does not depend on `page_*` being in `NextIntlClientProvider`.

### 5.3 Client components that only need `t()`

- For keys in **common / api** only, use **`useWarnTranslations()`**, consistent with `BootstrapsProvider` injecting `I18nService` (still ensure the layout’s messages include those keys).

### 5.4 `AppPageRouteParams` (optional)

- Some routes wrap `getI18nInterface` / `getI18nMessages` in `server/render/AppPageRouteParams.ts` for a single place to read `locale` from `params`.

## 6. Pages Router recommendations

### 6.1 `_app` injects `messages`

- `src/pages/_app.tsx` passes **`pageProps.messages`** into `NextIntlClientProvider`.
- Each page’s **`getStaticProps` / `getServerSideProps`** must return merged **`messages`**, usually via **`PagesRouteParams.getI18nMessages(namespace)`**.

### 6.2 `PagesRouteParams` summary

- Defined in `server/render/PagesRouteParams.ts`, **Node data-fetching only** (no `next-intl/server` request context).
- **`getI18nMessages(namespace?)`**: merges **`defaultNamespaces` (`common` + `api`)** with the page namespace, then **`loadMessages` + `filterMessagesByNamespace`**.
- **`getI18nInterface(mapping, namespace?)`**: builds a local `t` from those messages and `TranslateI18nUtil.translate`; can be used inside **`getStaticProps`** and returned in **`props`** for **fully static page copy** without client `useI18nMapping`.

### 6.3 With `PageI18nProvider`

- If the client uses **`useI18nMapping`** to build `seoMetadata`, wrap with **`PageI18nProvider value={seoMetadata}`** so children can use **`usePageI18nMapping`** (see `src/pages/[locale]/admin/request-logs.tsx`).
- For **build/request-time fixed copy**, prefer **`getI18nInterface` in `getStaticProps` + props**, then optionally still wrap with `PageI18nProvider`.

## 7. IOC and `I18nService`

- `src/uikit/components/BootstrapsProvider.tsx` in a client **`useEffect`**:
  - `changeLanguage(locale)` updates language state;
  - **`setTranslator(t)`** injects **`useWarnTranslations()`** into **`I18nService`**.
- Non-React code can resolve **`I18nServiceInterface`** from the container and call **`t(key)`** (after client mount and providers are ready).

## 10. Checklist when adding page copy

1. Add keys (or files) under **`i18n-identifier`** following namespace conventions.
2. Run **`generateLocales`** and verify **`public/locales`** for each locale (if JSON is not committed, ensure CI/local build runs generation).
3. Add a **readonly mapping object** and exported type under **`i18n-mapping`**.
4. **App Router:** resolve with **`getI18nInterface`** in page/layout and pass via **`PageI18nProvider`**, or ensure messages include the namespace before using **`useI18nMapping` / `useWarnTranslations`**.
5. **Pages Router:** in **`getStaticProps` / `getServerSideProps`**, call **`getI18nMessages(page namespace)`**; optionally **`getI18nInterface`** into **`props`**.
6. SEO: prefer **`getI18nInterface`** for metadata so it matches page language.

---

## 11. Common pitfalls

1. **Mixing “server full messages” with “client Provider messages”**  
   `request.ts` loads the full bundle; the root layout only passes **common + api** into `NextIntlClientProvider`. Page keys need a nested layout merge or **`PageI18nProvider`**.

2. **Using `useNextTranslations` in app code**  
   Use the project wrapper **`useWarnTranslations`** (the implementation imports `next-intl`’s `useTranslations` internally).

3. **Calling `usePageI18nMapping` without `PageI18nProvider`**  
   Throws; inject **`value={tt}`** from a parent Server/Client component.

4. **Pages route omitting `messages`**  
   `_app` expects **`pageProps.messages`**; without them, `next-intl` has no dictionary.
