# next-seed 国际化（i18n）说明

**English:** [i18n.en.md](./i18n.en.md)

本文说明 `next-seed` 中国际化的分层、数据流，以及在 **App Router** 与 **Pages Router** 下的推荐用法。实现上基于 [next-intl](https://next-intl-docs.vercel.app/)，并叠加项目内的 **标识层 → mapping 层 → 应用层** 约定。

---

## 1. 核心概念一览

| 概念                                          | 作用                                                                                                                                                                                          |
| --------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **词典（messages）**                          | `next-intl` 使用的键值表，键为带命名空间前缀的字符串（如 `common:save`、`api:user__not_found`）。                                                                                             |
| **默认命名空间**                              | `shared/config/i18n.ts` 中 `defaultNamespaces: ['common', 'api']`。根 layout 注入给客户端的 `NextIntlClientProvider` 的 messages **默认只包含** 这些命名空间（可与页面额外 namespace 合并）。 |
| **`PageI18nInterface`**                       | `shared/config/i18n-mapping/PageI18nInterface.ts` 定义的页面级文案/SEO 形状；具体页面的 mapping 对象（如 `homeI18n`）字段值为 **消息键**，经翻译后成为展示用字符串。                          |
| **`PageI18nProvider` + `usePageI18nMapping`** | 把「当前页已解析完的 mapping 对象」通过 React Context 传给子树，子组件直接读字符串，无需再调用 `t()`。                                                                                        |

---

## 2. 翻译流程简例（三种场景）

用一条线串起来：**标识层常量（消息键）→ mapping 对象（字段值 = 键）→ 页面或组件消费**。下面为示意代码，键名与文件路径按你实际页面替换即可。

### 2.1 App Router 页面

**流程**：`generateLocales` 写入 `public/locales` → 服务端 `getI18nInterface(locale, mapping, namespace)` 得到已翻译对象 `tt` → `<PageI18nProvider value={tt}>` → 子树客户端组件 `usePageI18nMapping()` 直接读 `tt.title` 等。

```tsx
// --- shared/config/i18n-identifier/pages/page.example.ts
// 导出消息键（运行 tools/generateLocales 后进入 en.json / zh.json）

// --- shared/config/i18n-mapping/ExamplePageI18n.ts
import {
  EXAMPLE_SAVE_KEY,
  EXAMPLE_TITLE_KEY
} from '../i18n-identifier/pages/page.example';

export const examplePageI18nNamespace = 'page_example';

export const examplePageI18n = Object.freeze({
  title: EXAMPLE_TITLE_KEY,
  saveButton: EXAMPLE_SAVE_KEY
});

export type ExamplePageI18nType = typeof examplePageI18n;

// --- src/app/[locale]/example/page.tsx（Server Component）
import { PageI18nProvider } from '@/uikit/context/PageI18nContext';
import {
  getI18nInterface,
  getLocale,
  type PageParamsType
} from '@server/render/pageRouteParams';
import {
  examplePageI18n,
  examplePageI18nNamespace
} from '@config/i18n-mapping/ExamplePageI18n';

export default async function ExamplePage({
  params
}: {
  params: Promise<PageParamsType>;
}) {
  const { locale: localeParam } = await params;
  const locale = getLocale({ locale: localeParam });
  const tt = await getI18nInterface(
    locale,
    examplePageI18n,
    examplePageI18nNamespace
  );

  return (
    <PageI18nProvider value={tt}>
      <ExamplePanel />
    </PageI18nProvider>
  );
}

// --- ExamplePanel.tsx（Client Component）
('use client');

import { usePageI18nMapping } from '@/uikit/context/PageI18nContext';
import type { ExamplePageI18nType } from '@config/i18n-mapping/ExamplePageI18n';

export function ExamplePanel() {
  const tt = usePageI18nMapping<ExamplePageI18nType>();
  return (
    <>
      <h1>{tt.title}</h1>
      <button type="button">{tt.saveButton}</button>
    </>
  );
}
```

**要点**：根 layout 给客户端的 messages 默认只有 `common` + `api`；页面专属键靠服务端 `getI18nInterface` 解析进 `tt`，子组件不依赖 `page_*` 是否打进 `NextIntlClientProvider`。

### 2.2 Pages Router 页面

**流程**：`getStaticProps` / `getServerSideProps` 里 **`new PagesRouteParams(params).getI18nMessages('page_example')`** → 得到 `common` + `api` + 页面 namespace → 作为 **`props.messages`** → `_app` 传入 `NextIntlClientProvider` → 页面（客户端）用 **`useI18nMapping(examplePageI18n)`** 得到 `tt`，或可选 **`getI18nInterface` 进 props** 完全在构建期/请求期定稿。

```tsx
// --- getStaticProps（节选）
import { PagesRouteParams } from '@server/render/PagesRouteParams';
import type { PagesRouteParamsType } from '@server/render/PagesRouteParams';
import type { GetStaticPropsContext } from 'next';

const namespace = 'page_example';

export async function getStaticProps({
  params
}: GetStaticPropsContext<PagesRouteParamsType>) {
  const pageParams = new PagesRouteParams(params);
  const messages = await pageParams.getI18nMessages(namespace);

  return {
    props: { messages }
  };
}

// --- src/pages/[locale]/example.tsx（Client Component 页面）
('use client');

import { useI18nMapping } from '@/uikit/hook/useI18nMapping';
import { examplePageI18n } from '@config/i18n-mapping/ExamplePageI18n';

export default function ExamplePage() {
  const tt = useI18nMapping(examplePageI18n);
  return <h1>{tt.title}</h1>;
}
```

**可选（更偏静态）**：在 `getStaticProps` 中 `await pageParams.getI18nInterface(examplePageI18n, namespace)`，把返回的 `tt` 放进 `props`，页面只展示、不再调用 `useI18nMapping`。

### 2.3 普通组件（两种路由下都能用）

| 场景                                       | 做法                                                                                                        |
| ------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| 文案在 **`common` / `api`**                | 客户端组件里 **`const t = useWarnTranslations(); return t(SOME_KEY);`**（与根 layout 默认注入的词典一致）。 |
| 文案属于 **当前页 mapping 且已由页面解析** | 组件必须挂在 **`PageI18nProvider`** 下，使用 **`usePageI18nMapping<T>()`** 读字段。                         |

```tsx
'use client';

import { useWarnTranslations } from '@/uikit/hook/useWarnTranslations';
// 自 identifier 导入的键，且属于 common / api 命名空间
import { SOME_COMMON_KEY } from '@config/i18n-identifier/common/common';

export function SaveChip() {
  const t = useWarnTranslations();
  return <span>{t(SOME_COMMON_KEY)}</span>;
}
```

（`SOME_COMMON_KEY` 仅为占位符，请换成 `i18n-identifier` 里 `common` 命名空间下的真实导出常量。）

**注意**：`useWarnTranslations` / `useI18nMapping` 只能在 **客户端**、且在 **`NextIntlClientProvider`** 已提供对应 `messages` 时使用；**服务端**请用 **`getI18nInterface`**，再把字符串经 props 或 `PageI18nProvider` 传给子组件。

## 3. 三层架构

### 3.1 第一层：标识层（`shared/config/i18n-identifier`）

- 存放 **稳定的消息键**，通常按模块拆成多个 `.ts` 文件。
- 键在生成的 JSON 里体现为 **`命名空间:键名`** 形式（与 `ts2locales` 生成规则一致）。
- **开发者不必手改** `public/locales/*.json`：由脚本从标识层生成。

### 3.2 第二层：Mapping 层（`shared/config/i18n-mapping`）

- 开发者日常接触的 **可读对象**：如 `homeI18n`、`adminRequestLogs18n`。
- 对象的 **属性值** 引用标识层导出的常量（即 next-intl 的 **message key**）。
- 注释与类型（如 `HomeI18nInterface`）说明该页/模块有哪些字段。

### 3.3 第三层：应用层

在组件或页面中，任选其一或组合使用：

1. **`useWarnTranslations()`**  
   返回增强后的 `t`，用于按 key 翻译（适合零散文案、或与 mapping 字段组合使用）。内部基于 `next-intl` 的 `useTranslations`，并套了缺失键时的告警/降级逻辑。

2. **`useI18nMapping(mappingObject)`**  
   在 **客户端** 把「整份 mapping（值为 key）」转成「整份已翻译对象」，适合 Pages 路由或没有服务端预解析的场景。

3. **`getI18nInterface(locale, mapping, namespace?)`（服务端）**  
   在 **服务端** 把 mapping 转成已翻译对象，用于 `generateMetadata`、Server Component 正文，或通过 `PageI18nProvider` 传给客户端子树。

4. **`usePageI18nMapping<T>()`**  
   读取 **父级 `PageI18nProvider`** 注入的已解析对象（类型 `T` 需继承/符合 `PageI18nInterface` 等约定）。

## 4. 词典如何生成与加载

### 4.1 生成：`tools/generateLocales.ts`

- 默认扫描目录：`shared/config/i18n-identifier` 下所有 `.ts`。
- 输出：`public/locales/{{lng}}.json`（如 `en.json`、`zh.json`），语言列表来自 `i18nConfig.supportedLngs`。
- 构建/本地脚本中执行生成即可；是否将 JSON 纳入版本控制由团队策略决定。

### 4.2 加载：`src/i18n/loadMessages.ts`

- **默认**从打包进来的 `public/locales/{locale}.json` 读取（`useApiLocales === false` 时）。
- 若 `shared/config/common.ts` 中 **`useApiLocales` 为 `true`**，会尝试从 `SITE_URL/api/locales/json?locale=...` 拉取，失败则回退到本地 JSON。

### 4.3 App Router 服务端配置：`src/i18n/request.ts`

- `getRequestConfig` 中对每个请求调用 **`loadMessages(locale)` 加载整包消息**（不按 namespace 裁剪）。
- 因此服务端 API（如 `getTranslations`）在 **一次请求内** 可解析任意存在于 JSON 中的 key。

### 4.4 客户端可见词典：根 `layout` 的裁剪

- `src/app/[locale]/layout.tsx` 使用 **`getI18nMessages(locale)`**（见 `server/render/pageRouteParams.ts`）：在 `getMessages` 得到全量后，按 **`i18nConfig.defaultNamespaces`（common + api）** 过滤，再交给 `NextIntlClientProvider`。
- **含义**：客户端 `useWarnTranslations()` / `useI18nMapping` 若要用 **页面专属 namespace**（如 `page_home`）的 key，必须保证对应 messages 已进入 Provider——做法是 **在更上层或本页 layout 合并该 namespace**，或使用下文 **`PageI18nProvider` 注入已解析文案**，避免客户端拿不到 key。

## 5. App Router 推荐做法

### 5.1 通用词典注入（已存在根 layout）

- `NextIntlClientProvider` 注入 **默认 common + api**，供全站通用组件、`I18nService.t` 等同源翻译使用。

### 5.2 页面级：服务端先解析再下传（首页模式）

典型步骤（参见 `src/app/[locale]/page.tsx`）：

1. `getLocale(resolvedParams)` 取当前语言。
2. **`const tt = await getI18nInterface(locale, pageMapping, pageNamespace)`** 得到整页已翻译对象。
3. **`generateMetadata`** 中可对 SEO 使用 **`await getI18nInterface(locale, pageMapping)`**（是否传第三个参数依该页 metadata 是否依赖特定 namespace 而定）。
4. 用 **`<PageI18nProvider value={tt}>`** 包裹本页客户端子树；深层客户端组件使用 **`usePageI18nMapping<YourPageI18nType>()`** 读取 `tt` 的字段。

**优点**：页面文案在服务端一次性解析；客户端不依赖 `page_*` 是否打进 `NextIntlClientProvider`。

### 5.3 仅用 `t()` 的客户端组件

- 在 **仅依赖 common / api** 的键时，直接使用 **`useWarnTranslations()`**，与 `BootstrapsProvider` 注入 `I18nService` 的 `t` 行为一致（同一套 messages 来源逻辑需注意 layout 是否包含该 key）。

### 5.4 `AppPageRouteParams`（可选）

- 部分页面通过 `server/render/AppPageRouteParams.ts` 封装 `getI18nInterface` / `getI18nMessages`，与 `pageRouteParams` 行为对齐，便于统一从 `params` 取 `locale`。

## 6. Pages Router 推荐做法

### 6.1 `_app` 注入 messages

- `src/pages/_app.tsx` 将 **`pageProps.messages`** 传给 `NextIntlClientProvider`。
- 各页的 **`getStaticProps` / `getServerSideProps`** 必须返回合并后的 **`messages`**，且通常通过 **`PagesRouteParams.getI18nMessages(namespace)`** 获取。

### 6.2 `PagesRouteParams` 行为摘要

- 定义于 `server/render/PagesRouteParams.ts`，**仅在 Node 数据获取阶段使用**（不依赖 `next-intl/server` 的请求上下文）。
- **`getI18nMessages(namespace?)`**：合并 **`defaultNamespaces`（common + api）** 与传入的页面 namespace，再 **`loadMessages` + `filterMessagesByNamespace`**。
- **`getI18nInterface(mapping, namespace?)`**：基于上述 messages 构造本地 `t`，用 `TranslateI18nUtil.translate` 得到已解析对象；可在 **`getStaticProps` 里把结果放进 `props`**，实现 **纯静态的页面文案**，避免客户端再跑 `useI18nMapping`。

### 6.3 与 `PageI18nProvider` 组合

- 若在客户端用 **`useI18nMapping`** 得到 `seoMetadata`，可 **`PageI18nProvider value={seoMetadata}`** 供子组件 **`usePageI18nMapping`** 使用（参见 `src/pages/[locale]/admin/request-logs.tsx` 一类页面）。
- 文档推荐：若追求 **构建期/请求期固定文案**，优先在 **`getStaticProps` 中 `getI18nInterface` + props 传递**，再视需要挂 `PageI18nProvider`。

---

## 7. 与 IOC / `I18nService` 的关系

- `src/uikit/components/BootstrapsProvider.tsx` 在客户端 **`useEffect`** 中：
  - `changeLanguage(locale)` 更新语言状态；
  - **`setTranslator(t)`** 将 **`useWarnTranslations()`** 返回的 `t` 注入 **`I18nService`**。
- 非 React 路径可通过容器取得 **`I18nServiceInterface`**，调用 **`t(key)`** 使用与界面同一套翻译函数（需已在客户端挂载且 Provider 已就绪）。

## 8. 工具类与 hook 对照

| 符号                                      | 路径/位置                               | 用途                                                                              |
| ----------------------------------------- | --------------------------------------- | --------------------------------------------------------------------------------- |
| `useWarnTranslations`                     | `src/uikit/hook/useWarnTranslations.ts` | 组件内 `t('key')`；对缺失键做告警或降级（受 `i18nWarnMissingTranslation` 影响）。 |
| `useI18nMapping`                          | `src/uikit/hook/useI18nMapping.ts`      | 客户端将 mapping 对象整表翻译。                                                   |
| `PageI18nProvider` / `usePageI18nMapping` | `src/uikit/context/PageI18nContext.tsx` | 注入/读取「当前页已解析」的 mapping。                                             |
| `getI18nInterface` / `getI18nMessages`    | `server/render/pageRouteParams.ts`      | App Router 静态辅助（配合 `getTranslations` / `getMessages`）。                   |
| `PagesRouteParams`                        | `server/render/PagesRouteParams.ts`     | Pages 数据获取阶段加载 messages / 解析 interface。                                |
| `TranslateI18nUtil`                       | `src/impls/TranslateI18nUtil.ts`        | 对「值为 string 的扁平 mapping」逐键调用 `t`。                                    |
| `i18nConfig`                              | `shared/config/i18n.ts`                 | `defaultNamespaces`、`supportedLngs`、`fallbackLng` 等。                          |

## 9. 配置项速查

| 配置                         | 文件                      | 说明                                                                                                                  |
| ---------------------------- | ------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| `defaultNamespaces`          | `shared/config/i18n.ts`   | 默认 `common`、`api`。                                                                                                |
| `useApiLocales`              | `shared/config/common.ts` | 是否优先从 API 拉取词典。                                                                                             |
| `i18nWarnMissingTranslation` | `shared/config/common.ts` | 是否在 `useWarnTranslations` 路径上对缺失翻译打日志而非直接抛错（与 `TranslateI18nUtil.overrideTranslateT` 等配合）。 |

## 10. 新增一页文案时的检查清单

1. 在 **`i18n-identifier`** 增加键（或新文件），保证命名空间与键名符合现有约定。
2. 运行 **`generateLocales`**，确认 **`public/locales`** 中各语言条目已更新（若未提交 JSON，则 CI/本地构建需包含生成步骤）。
3. 在 **`i18n-mapping`** 为对应页面/模块增加 **readonly 对象** 与类型导出。
4. **App Router**：在 **page** 或 **layout** 用 **`getI18nInterface`** 解析并 **`PageI18nProvider`** 下传，或在确保 messages 含该 namespace 后用 **`useI18nMapping` / `useWarnTranslations`**。
5. **Pages Router**：在 **`getStaticProps`（或 `getServerSideProps`）** 中调用 **`getI18nMessages(页面 namespace)`**，必要时 **`getI18nInterface`** 写入 **`props`**。
6. SEO：metadata 字段优先走 **`getI18nInterface`**，保证与页面正文语言一致。

## 11. 常见误区

1. **混淆「服务端全量 messages」与「客户端 Provider messages」**  
   `request.ts` 加载整包；根 layout 只把 **common + api** 裁进 `NextIntlClientProvider`。页面专属 key 需在子 layout 合并 namespace 或用 **`PageI18nProvider`**。

2. **在客户端查找 `useNextTranslations`**  
   业务代码应使用项目封装的 **`useWarnTranslations`**（实现内部才引用 `next-intl` 的 `useTranslations`）。

3. **未包裹 `PageI18nProvider` 即调用 `usePageI18nMapping`**  
   会抛错；请在上层 Server/Client 组件注入 **`value={tt}`**。

4. **Pages 路由忘记返回 `messages`**  
   `_app` 依赖 **`pageProps.messages`**；未返回则 next-intl 无词典可用。
